Specifically, during the run-time dynamic linker (`rtld`) stage, this manipulation grants both an Arbitrary Address Write (`AAW`) primitive and an Arbitrary Address Call primitive.

- Armed with these two primitives, I was able to construct and implement a fully functioning, Turing-complete machine that executes entirely at load time. Achieving this required actively exploiting the dynamic linker (`ld.so`) itself. Consequently, the massive block of Elf64_Rela entries found in the binary are not standard linking instructions at all; they act as a sophisticated payload of exploit code targeting ld.so. It is during the execution of this weaponized load-time code that the actual, hidden flag-checking logic is evaluated.

- Finally, a critical note regarding the extraction methodology: if you attempted to extract the underlying binary using purely static techniques, it is highly likely you pulled the wrong file. This is because the main binary's malicious `RELA` entries are actively patching other secondary binaries that are embedded directly within global variables. Because this patching process alters the executable in memory, you absolutely must extract the binary dynamically to ensure you are analyzing the fully modified, correct version.
